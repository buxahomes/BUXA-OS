# AGENTS.md

# BUXA Operating System Repository

This repository contains the official BUXA Operating System.

BUXA OS is the authoritative operating environment for BUXA business knowledge, brand governance, product information, project delivery systems, content operations, marketing, artificial intelligence and automation.

Every AI agent working inside this repository must treat the repository as the Single Source of Truth for BUXA.

Agents must strengthen the system rather than produce isolated outputs.

---

# 1. Purpose

The purpose of this file is to define how AI agents work inside BUXA OS.

It governs:

- repository navigation;
- knowledge retrieval;
- task classification;
- document creation;
- content generation;
- file modification;
- quality control;
- approval boundaries;
- Git operations;
- cross-module consistency.

This file does not replace official business knowledge.

It defines how agents must use and maintain that knowledge.

---

# 2. Mission

Every agent working in this repository has one primary responsibility:

> Strengthen BUXA OS as a coherent, reusable and maintainable operating system.

Do not create isolated solutions that cannot be reused.

Every contribution should improve one or more of the following:

- clarity;
- consistency;
- correctness;
- maintainability;
- reusability;
- governance;
- traceability;
- execution quality;
- knowledge quality.

Completion means improving the system, not merely producing an output.

# 3. System Identity

BUXA OS supports the long-term operating principle:

> One Brand. One Knowledge Base. One Operating System.

BUXA OS is intended to become:

- the official operating system of BUXA;
- the Single Source of Truth for business knowledge;
- the foundation for an AI workforce;
- the foundation for repeatable workflows;
- the foundation for automation;

# 4. Authority Hierarchy

Before making any decision, understand the hierarchy of authority.

```text
People
    │
    ▼
Official Business Knowledge
    │
    ▼
Business Governance
    │
    ▼

    │

Authority always flows from top to bottom.

Artificial intelligence must never override:

approved human decisions;
official business knowledge;
governance rules;
approved product data;
approved brand positioning;
legal or commercial policies.

Automation executes approved rules.

It does not create authority.

#27. Change Levels

Classify repository changes before implementation.

Level 1 — Low-risk maintenance

Examples:

correcting grammar;
fixing formatting;
repairing broken links;
adding missing cross-references;
updating indexes from approved content;
normalising filenames under an approved standard.

Agents may generally perform these changes.

Level 2 — Structured operational change

Examples:

adding a content template;
creating a draft content asset;
adding an approved workflow;
extending an existing module;
adding metadata;
adding a platform checklist.

Agents may prepare and implement these changes when the task is explicit and existing architecture is preserved.

Level 3 — Knowledge or governance change

Examples:

changing product specifications;
changing PDMS responsibilities;
changing brand positioning;
changing business rules;
changing governance;
introducing a new official term;
changing repository architecture;
deleting official documentation.

These changes require human approval before implementation.

#29. File Modification Rules

Before modifying a file:

read the complete relevant section;
understand its purpose;
check related files;
identify whether it is authoritative;
preserve approved knowledge;
minimise the scope of change;
avoid unrelated edits;
validate cross-references.

When replacing a file:

preserve all valid existing knowledge;
explain material changes;
avoid silent deletion;
maintain compatible terminology;
retain historical context where necessary.

Do not modify multiple authoritative files merely to make them appear stylistically uniform.

#30. Git Safety Rules

Before making changes:

confirm the repository;
confirm the current branch;
check git status;
confirm the working tree state;
inspect existing changes;
avoid overwriting uncommitted human work.

Preferred workflow for substantial changes:

inspect
→ plan
→ create branch
→ modify
→ validate
→ review diff
→ commit
→ push
→ open pull request

Do not push directly to main unless explicitly authorised.

Do not force-push unless explicitly authorised.

Do not delete branches or rewrite history without explicit approval.

Do not include unrelated files in a commit.
